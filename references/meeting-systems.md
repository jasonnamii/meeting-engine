# Meeting Systems — 정기 회의 체계 설계 & 감사

정기 회의("우리는 매주 월요일 오전 10시에 모입니다")는 대부분 무사고적으로 존재한다. 즉, 필요 없이 반복되거나, 구조가 낙후되어 있거나, 리더가 바뀌면서 형식만 남음.

---

## Standing Meeting Audit Protocol (정기 회의 감시 프로토콜)

**목표**: "이 회의는 계속해야 하나? 폐지해야 하나? 재설계해야 하나?" 명확히 하기.

### 감사 질문지 (체크리스트)

**회의 기본**:

| 질문 | YES | NO | 개선안 |
|------|-----|----|----|
| 이 회의의 목적을 명확히 말할 수 있는가? | | | 목적 문서화 필수 |
| 그 목적이 정말 필요한가? (3개월 내 그 목적 때문에 의사결정을 했는가?) | | | 폐지 검토 |
| 참석자 전원이 "왜 내가 참석해야 하는지" 아는가? | | | 롤 정의 필수 |
| 아젠다가 24시간 전에 배포되는가? | | | 아젠다 배포 의무화 |
| 회의 시간이 지켜지는가? (±5분) | | | 타이머 도입 |

**의사결정 & 액션**:

| 질문 | YES | NO | 개선안 |
|------|-----|----|----|
| 이 회의에서 의사결정이 일어나는가? (지난 3회 회의에서 최소 1개의 결정이 기록됨) | | | 의사결정 권한 명확화 또는 회의 목적 변경 |
| 결정 사항이 회의록에 명시적으로 기록되는가? | | | 회의록 템플릿 강제 |
| 액션 아이템이 Who+What+When으로 명확한가? | | | 액션 아이템 검증 프로세스 추가 |
| 지난 회의의 액션 추적을 하는가? (완료/미완료/블로커 확인) | | | 액션 추적표 도입 + 오프닝 5분 |
| 미완료 액션에 대해 근인을 파고 있는가? | | | 매월 1회 "왜 액션이 안 끝났나?" 분석 |

**효율성**:

| 질문 | YES | NO | 개선안 |
|------|-----|----|----|
| 회의 시간이 필요한 것인가? (이 내용이 비동기 문서/이메일/슬랙으로도 가능한가?) | | | Async vs Sync Decision Framework 적용 |
| 회의 시간이 적절한가? (목적 대비 너무 길거나 짧지 않은가?) | | | 시간 재설계 |
| 불필요한 참석자가 있는가? | | | 참석 권장→선택으로 변경. 비동기 공유로 대체 |
| 회의 후 문서(회의록)를 정리하는가? | | | 회의록 5분 내 완성 의무화 |

### 감사 결과 해석

**YES 응답이 12개 이상**: 
- 회의는 건강합니다. 현재 체계 유지.
- 단, 분기마다 1회 감사 반복 (조직 변화 감지).

**YES 응답이 8~11개**:
- 개선 사항 2~3개 있음.
- 우선순위: ① 목적 명확화 → ② 아젠다 배포 → ③ 회의록 작성

**YES 응답이 5~7개**:
- 재설계 필요. 아래 "회의 시스템 재설계 프로토콜" 실행.

**YES 응답이 5개 이하**:
- 폐지 또는 완전 재구성 검토.

## Async vs Sync Decision Framework

**언제 동기식(회의)이 필요한가? 언제는 비동기(문서/슬랙)로 충분한가?**

### Decision Tree

```
Q1: 의사결정이 필요한가?
├─ NO → 비동기 정보공유 (Loom, 문서, 슬랙)
└─ YES:
   Q2: 논의(질의·답변·이견)가 필요한가?
   ├─ NO → 비동기 안내 (문서 배포 + 구글폼 의견수집)
   └─ YES:
      Q3: 실시간(토론 중 변경)이 필요한가?
      ├─ NO → 비동기 라운드 (구글닥 코멘트 기반 토론. 2~3일 진행)
      └─ YES → 동기식 회의 필수
```

### 상황별 판단

| 상황 | 필요 | 추천 형식 |
|------|------|---------|
| 지표 공유 (주간 매출, 사용자수 등) | 비동기 | 대시보드 자동 공유 + Slack 공지 |
| 팀 활동 리포트 | 비동기 | 문서 (매주 금요일 업데이트) |
| 의사결정 (예: 신기능 우선순위) | 동기 | 회의 (발제자 준비 필수) |
| 브레인스토밍 | 동기 | 회의 + Nominal Group Technique |
| 피드백 (개인 성과 평가) | 동기 | 1:1 (30분) |
| 교육/학습 | 비동기 | 온라인 강의 + 선택적 Q&A 회의 |
| 긴급 이슈 | 동기 | 즉각 회의 또는 온라인 채팅 |
| 깊은 전략 의사결정 | 동기 | Amazon Memo 방식 (30분 독서 + 토론) |

### 비동기 형식 가이드

**구글 문서 + 코멘트 기반 토론**:
```
1일차: 발제자가 문서 작성 → 팀 공유
2~3일차: 팀원이 각자 코멘트 (의견/질문)
4일차: 발제자가 코멘트 정리 + 결정
5일차: 최종 결과 공시
```

**Loom 비디오 + Slack 스레드**:
```
1. 발제자가 2~3분 Loom 비디오 녹화
2. Slack에서 video 공유 + "의견은 스레드에"
3. 팀원이 스레드에서 질문/의견
4. 발제자가 스레드에서 답변
5. 필요시 follow-up 회의
```

**Google Forms 투표**:
```
1. 3개 선택지 제시 + "왜 이 선택지를 권하나" 설명
2. 익명 투표 (24시간)
3. 결과 공시 + 가장 많은 투표 받은 것 선택
```

**언제 비동기로는 부족한가?**
- 실시간으로 의견이 충돌하고 즉각 통합안이 필요할 때
- 복잡한 질문이 많이 나올 것 같을 때 ("이 부분이 뭔 말이야?")
- 리더가 즉각 판단해야 할 때 (비동기는 지연 발생)
- 팀의 심리안전이 낮아 비공개(1:1)가 필요할 때

## Scrum Ceremonies Best Practices

**Agile 팀의 정기 회의 4가지. 각각의 목적·시간·참석자·구조 명확히.**

### 1. Sprint Planning (스프린트 계획)

| 항목 | 표준 | 주의 |
|------|------|------|
| **시간** | 총 스프린트 길이의 5% (2주 = 2시간) | 3시간 이상 절대 금지 (피로) |
| **참석** | 스크럼팀 + 제품 리더 (PO) | PO가 없으면 반쪽짜리 회의 |
| **목표** | 스프린트 동안 "뭘 할 건가" 정의 | "얼마나 많은 일을 할 건가" 다 추정하려 하면 안 됨 |
| **구조** | ① PO가 백로그 우선순위 설명 ② 팀이 질문 ③ 팀이 자신의 역량에 맞춰 선택 ④ 스프린트 목표 1문장 정의 | "우리 스프린트의 목표는 X를 완성하는 것" 명확히 |

**흔한 실수**:
- "예상 추정치"를 기반으로 일감 정하기 (실제와 달라짐)
- PO가 "이것도 하고 저것도 해"라고 강요
- 스프린트 목표가 없음 ("아, 백로그에 있는 거 하면 되지")

**개선안**:
- 추정이 아니라 "팀의 과거 처리량(Velocity)"에 기반
- PO는 "왜" 우선순위인지만 설명. "뭘" 할지는 팀이 결정
- "이번 스프린트의 가장 중요한 하나는 X입니다" 선언

### 2. Daily Standup (일일 스탠드업)

| 항목 | 표준 | 주의 |
|------|------|------|
| **시간** | 정확히 15분 | 30분 넘으면 망한 것 |
| **형식** | 각자 "어제 뭘 했나 / 오늘 뭘 할 건가 / 블로커 있나" (1분씩) | "상세 보고" 하는 관리자 발상 금지 |
| **목적** | 팀이 서로를 파악 + 블로커 즉시 해결 | "리더에게 상태 보고하기" 아님 |
| **참석** | 스크럼팀 + 선택적으로 이해관계자 | 이해관계자가 많으면 팀이 말 안 함 |

**흔한 실수**:
- 리더가 "왜 이걸 못 했어?" 물어보기 (탓하기)
- 상세 기술 토론 시작 ("코드 리뷰 기준이...")
- 서 있지 않고 앉아서 진행 (시간 길어짐)

**개선안**:
- "블로커 해결"만 담당. 상세 논의는 회의 후 따로
- "왜" 물어보는 대신 "뭘 도와줄까?" 물어보기
- 물리적으로 서서 진행 (자연히 짧아짐)

### 3. Sprint Review (스프린트 리뷰 / Demo)

| 항목 | 표준 | 주의 |
|------|------|------|
| **시간** | 1시간 (2주 스프린트 기준) | 준비 없는 라이브 데모 금지 |
| **참석** | 팀 + 모든 이해관계자 + 고객 (가능하면) | 리더만 있으면 피드백 편향 |
| **목표** | "우리가 뭘 완성했는가" **보여주기** | 보고서 작성 아님 |
| **구조** | ① 스프린트 목표 상기 ② 완성된 것 **데모** ③ 피드백 수집 | 완성 안 된 것은 보여주지 말 것 |

**흔한 실수**:
- 스크린샷으로만 설명 (실제 사용감 없음)
- 고객이 없음 ("우리끼리만")
- 완성 안 된 것도 섞음 ("이건 거의 다 됐어")

**개선안**:
- 실제로 사용해보게 함 (고객이 버튼을 클릭하게)
- 고객/이해관계자의 반응 적극 수집
- "Done"의 기준을 스프린트 시작 전에 정의

### 4. Sprint Retrospective (스프린트 회고)

| 항목 | 표준 | 주의 |
|------|------|------|
| **시간** | 45분 (2주 기준) | 1시간 절대 금지 (지침) |
| **참석** | 스크럼팀 (리더는 선택적) | 리더가 있으면 팀이 솔직하게 말 안 함 |
| **목표** | "**우리**가 **뭘 잘했고 뭘 개선할 건가**" 스스로 파악 | "리더에게 문제 보고하기" 아님 |
| **구조** | ① 긍정: 뭘 잘했나? ② 개선: 뭘 개선할까? ③ 액션: 다음 스프린트에서 뭘 할 건가? | Start/Stop/Continue 템플릿 권장 |

**Start/Stop/Continue 템플릿**:
```
START: "우리가 시작해야 할 것"
- 일일 스탠드업을 아침 9시로 옮기자 (현재 10시)
- 코드 리뷰 체크리스트를 만들자

STOP: "우리가 멈춰야 할 것"
- 스프린트 중간에 긴급 요청받기 (계획 깨짐)
- 회의를 너무 많이 하기

CONTINUE: "우리가 계속 해야 할 것"
- 주 2회 peer 코드 리뷰(잘 작동)
- 스프린트 리뷰에 고객 초대 (피드백 좋음)
```

**흔한 실수**:
- "더 열심히 하자" 같은 일반화된 피드백
- 리더가 문제 원인을 분석해주기 ("그건 OOO 때문이지")
- 액션이 없음 ("그래, 그렇게 하자" 끝)

**개선안**:
- 구체적 행동 액션만: "일일 스탠드업을 아침 9시로 옮긴다" (WHO이 아니면 리더 담당)
- Scrum Master 또는 별도 진행자가 리더 없이 진행
- 액션을 다음 스프린트 계획에 반영 (Done이 되는지 추적)

## Sprint Metrics & Velocity Tracking

**팀의 처리량 파악하기**:

| 지표 | 정의 | 계산 | 사용 |
|------|------|------|------|
| **Velocity** | 팀이 1 스프린트에 완성하는 스토리 포인트 | 지난 3~4 스프린트 평균 | 다음 스프린트 용량 예측 |
| **Burndown** | 스프린트 동안 남은 일의 양 | 일일 기준. 목표: 일직선으로 내려감 | 스프린트 진행도 추적 |
| **Cycle Time** | 작업을 시작한 후 완성까지 걸린 시간 | Jira의 "In Progress" → "Done" | 과정 개선 (병목은 뭔가?) |
| **Cumulative Flow** | 백로그→진행중→완성의 흐름 | 시간에 따른 각 단계 아이템 수 | 팀 용량 편향 파악 |

**주의**:
- Velocity를 "팀의 성과"로 평가하면 안 됨 (팀이 수치 맞추려고 스토리 포인트 부풀림)
- Velocity는 "용량 예측 도구"일 뿐, "성과 평가"가 아님

## Standing Meeting Types Checklist

**당신의 조직에서 필요한 정기 회의는?**

| 회의명 | 빈도 | 시간 | 목적 |
|--------|------|------|------|
| **일일 스탠드업** | 매일 | 15분 | 팀 동기화 + 블로커 해결 |
| **주간 진척 리뷰** | 매주 | 45분 | 지난주 액션 추적 + 이번주 계획 |
| **월간 전략 회의** | 매월 | 2시간 | 조직 목표·리소스·리스크 리뷰 |
| **분기별 OKR 리뷰** | 분기마다 | 3시간 | OKR 달성도 평가 + 다음 분기 계획 |
| **스프린트 계획** | 스프린트 시작 | 2시간 (2주 기준) | 스프린트 백로그 정의 |
| **스프린트 리뷰** | 스프린트 종료 | 1시간 | 완성된 것 데모 + 피드백 |
| **스프린트 회고** | 스프린트 종료 | 45분 | 프로세스 개선 액션 도출 |
| **1:1** | 주 1회 | 30분 | 개인 피드백·커리어·신뢰 구축 |
| **팀 빌딩** | 분기마다 | 2시간 | 팀 심리안전·응집력 |

**간단한 규칙**: 
- 동기식 회의는 최소화 (최대 주 4회)
- 모든 정기 회의는 3개월마다 "필요한가?" 재점검
- "우리 회의가 너무 많아" 느끼면 위의 Standing Meeting Audit 실행
