# Facilitation Science — 퍼실리테이션 과학 & 기법 모음

좋은 설계도 형편한 퍼실리테이션으로 망가진다. 반대도 마찬가지. 설계 단계에서 어떤 퍼실리테이션 패턴을 쓸지 결정해야 함.

---

## Kaner's Diamond of Participation

**저자**: Sam Kaner (Facilitator's Guide to Participatory Decision-Making, 1996)

**핵심**: 그룹이 합의에 도달하려면 3개의 뚜렷한 단계를 거쳐야 하고, 대부분의 퍼실리테이션 실패는 **"Groan Zone에서 너무 빨리 탈출하려고 할 때"** 발생.

**3단계 다이아몬드**:

```
Divergent Zone (발산)
    ↓ 아이디어 급증, 의견 차이 커짐
Groan Zone (고민존)  ← ★ 여기가 중요
    ↓ 불편함, 모순, 복잡함을 견디는 구간
Convergent Zone (수렴)
    → 합의에 도달
```

### 단계별 상세

**① Divergent Zone (10~20분)**

| 측면 | 특징 |
|------|------|
| **그룹의 심리 상태** | 신남. 아이디어 쏟아짐. "오, 이런 방법도 있네!" |
| **보이는 현상** | 발언 많음. 다양한 의견. 에너지 높음. |
| **퍼실리테이터의 역할** | 모든 아이디어 수용. 비판 금지. "더 있나?" 계속 채근. |
| **하지 말아야 할 것** | 평가·판단·논의 시작 X. "그건 이미 시도했는데..." X. |
| **신호: 끝났는가?** | 아이디어가 더 나오지 않을 때. 침묵이 30초 이상. |

**②Groan Zone (20~40분)**

| 측면 | 특징 |
|------|------|
| **그룹의 심리 상태** | 불편함. 혼란. "뭔가 복잡하네." "이게 맞나?" |
| **보이는 현상** | 침묵 증가. 깊은 신음("아..."). 몸 움직임(팔짱, 쓸어내림). 온도 내려감. |
| **퍼실리테이터의 역할** | 침묵을 견디기. "이 불편함이 정상입니다" 안심시키기. 질문으로 깊게 파기. |
| **하지 말아야 할 것** | 너무 빨리 수렴 방향 제시 X. "자, 이제 이것으로 가자" X (너무 빨리). |
| **함정** | **대부분의 리더가 여기서 탈출하고 싶어함**. "우리는 합의 못 하겠네"라고 결론짓고 투표로 넘어감. |

**③ Convergent Zone (10~20분)**

| 측면 | 특징 |
|------|------|
| **그룹의 심리 상태** | 안도감. "아, 이렇게 하면 되겠네." 에너지 다시 높아짐. |
| **보이는 현상** | "그럼 이렇게 하면?" "좋은 아이디어네" 발언 증가. 핵심으로 수렴. |
| **퍼실리테이터의 역할** | 수렴 흐름 가속화. "그래, 이 방향 맞네" 확인. 결정 강화. |
| **신호: 끝났는가?** | 모두가 고개를 끄덕일 때. 에너지가 안정됨. |

### Groan Zone 탈출 실패 사례

**시나리오**: 신제품 출시 여부 결정

```
① Divergent (10분)
"여름 출시해야 한다" "아니다 가을이 낫다" "웨비나로 테스트?" "유튜브 광고는?" 
→ 아이디어 30개

② Groan Zone 들어감
"근데 이게 다 가능해?" "예산은?" "개발 일정은?" 
침묵... 긴장... 

❌ 리더가 말함: "자, 우리는 여름 출시로 가자"
→ 팀이 생각한 게 아니라 리더가 정한 것
→ Convergent Zone 단계를 건너뜀
→ 결과: "그래서 뭘로 결정한 거야?" 명확성 부족

⭕ 대신 퍼실리테이터가 해야 할 말: 
"지금 복잡해 보이지만, 이게 정상입니다. 
자, 이 세 가지 쟁점(일정·예산·시장)을 하나씩 풀어볼까요?"
→ 팀이 Groan Zone을 견디면서 스스로 해결책을 찾음
→ Convergent Zone에 도달
→ 결과: "우리가 이렇게 결정했어" 주인의식
```

### Groan Zone을 견디는 퍼실리테이션 기술

| 기법 | 구체적 사용법 |
|------|------------|
| **침묵을 견디기** | "지금 침묵이 필요한 순간입니다. 30초만 더요." |
| **질문으로 파기** | "예산 문제는 어떻게 해결할까?" "누가 그걸 담당할 수 있을까?" |
| **작은 그룹으로 나누기** | 4명씩 3개 그룹 → 각각 쟁점을 풀기 → 모임 |
| **시각화** | 화이트보드에 "쟁점 1", "쟁점 2", "쟁점 3" → 하나씩 해결 |
| **비유/사례** | "예전에 비슷한 상황에서 우리가..." |
| **호기심 자극** | "만약 예산이 충분하다면?" "기술적으로 가능할까?" |

## Liberating Structures (33가지)

**저자**: Keith McCandless & Henri Lipmanowicz

**핵심**: Liberating Structures는 "어떻게 대화를 구조화할 것인가"의 33가지 패턴. 모두 10~40분 내 실행 가능.

### 분류별 구조

#### 1. Ideation & Divergence (발산·아이디어 생성)

| 이름 | 시간 | 진행 | 사용 시점 |
|------|------|------|---------|
| **1:1 Interviews** | 20분 | A↔B 1분, B↔C 1분... 모두가 다른 사람과 1분씩 대화 | 팀 이해 도입부 |
| **Appreciative Interviews** | 20분 | "우리가 잘한 것"을 1:1 면담으로 캐내기 | 긍정적 접근. 팀 사기 |
| **Brainstorm Web** | 20분 | 중앙 주제에서 연결로 뻗어나감. 다이어그램 방식 | 관계도 파악 |
| **Concept Fan** | 20분 | 하나의 해결책에서 가지치기로 변형안 도출 | 옵션 확장 |
| **Nine Whys** | 20분 | "왜?"를 9번 반복하면서 근인 파고들기 | 진짜 문제 찾기 |
| **Improv Prototyping** | 25분 | 제약 없이 역할극으로 아이디어 테스트 | 신제품 아이디어 |

**추천**: 브레인스토밍 시에는 반드시 **1:1 Interviews** 또는 **Appreciative Interviews** 먼저.

#### 2. Strategy & Alignment (전략·정렬)

| 이름 | 시간 | 진행 | 사용 시점 |
|------|------|------|---------|
| **Shift & Share** | 20분 | 좌석을 바꾸며 여러 테이블을 돌며 의견 수집 | 다양한 관점 수렴 |
| **Draw Toast** | 20분 | 토스트 만드는 과정을 그려봄 → 프로세스 이해 | 내 일의 상호 연결성 깨닫기 |
| **Ecocycle Planning** | 30분 | 성장/안정/쇠퇴/갱신 4단계에 아이디어 분류 | 우선순위 정렬 |
| **Social Network Webbing** | 30분 | 조직의 실제 영향력 관계도 그리기 | 조직 구조 vs 실제 권력 차이 |

#### 3. Decision Making (의사결정)

| 이름 | 시간 | 진행 | 사용 시점 |
|------|------|------|---------|
| **Wise Crowds** | 20분 | 익명 투표 + 결과 공개 + 재투표 | 신뢰도 기반 의사결정 |
| **Sticky Note Prioritization** | 25분 | 포스트잇에 의견/아이디어 적고 투표로 순위 | 빠른 의사결정. 참여도 높음 |
| **Generative Relationships Index** | 15분 | 팀 내 신뢰도/심리안전도 측정 후 개선점 논의 | 팀 건강도 진단 |

#### 4. Accountability & Progress (책임·진척)

| 이름 | 시간 | 진행 | 사용 시점 |
|------|------|------|---------|
| **Conversation Cafes** | 45분 | 작은 테이블에서 깊이 있는 대화 | 조직 전체 중요 주제 토론 |
| **Fishbowl** | 30분 | 원 가운데 4명이 대화, 밖의 사람들이 관찰 | 다양한 관점 공유 |
| **Min Specs** | 25분 | "최소 필수 요소"만 정의. 나머지는 팀 재량 | 권한위임 + 창의성 |

### 사용 Checklist

**브레인스토밍 회의**:
```
①Appreciative Interviews (20분: "이전에 잘한 것?")
→②Brainstorm Web (20분: 자유롭게 아이디어)
→③Sticky Note Prioritization (15분: 투표로 TOP 5)
```

**의사결정 회의**:
```
①선택지 3개 제시
→②Wise Crowds (익명 투표)
→③결과 공개 후 논의
→④재투표
```

**진척 리뷰**:
```
①Conversation Cafe (테이블 3개, 각 주제 담당)
→②각 테이블 대표 보고
→③Sticky Note로 blocker 분류 + 우선순위
```

## ToP ORID Model (Technology of Participation)

**저자**: Arnold & Donna Spencer (Institute of Cultural Affairs)

**핵심**: 의사결정의 4단계. 각 단계마다 구체적 질문 프로토콜이 있음.

**4단계**:

### ① Objective (객관적 사실)

**목적**: 모두가 같은 "사실"을 공유하기.

| 진행 | 퍼실리테이터 역할 | 예시 질문 |
|------|-----------------|---------|
| 라운드로빈 | 모두가 한 번씩 발언 (비판 없음) | "어제 회의에서 뭘 들었나요?" "어떤 데이터를 봤나요?" |
| 기록 | 보드에 모든 답 기록 (수정 금지) | |
| 체크 | "더 있나?" | |

**함정**: "아, 그건 틀렸어" 하면서 수정하면 안 됨. 객관적 사실만 수집.

### ② Reflective (감정·반응)

**목적**: 각자가 그 사실에 대해 어떤 **감정**을 느꼈는가.

| 진행 | 퍼실리테이터 역할 | 예시 질문 |
|------|-----------------|---------|
| 라운드로빈 | 모두가 감정 표현 (논리 X) | "어떤 기분이 들었나요?" "무엇에 놀랐나요?" |
| 수용 | "그런 감정도 있군요" 받아주기 | |
| 다양성 존중 | 같은 사실도 다양한 감정 가능 | |

**예시**:
- A: "불안했어요. 일정이 너무 타이트"
- B: "흥미로웠어요. 새로운 기술 배웠어"
- C: "답답했어요. 왜 이제서야 알려줬지?"

모두 대면적 감정이지만, 모두 정상. 평가 금지.

### ③ Interpretive (의미·해석)

**목적**: 그 감정들에서 나오는 **의미·인사이트**.

| 진행 | 퍼실리테이터 역할 | 예시 질문 |
|------|-----------------|---------|
| 깊은 질문 | "이게 왜 중요한가?" "이게 우리에게 뭘 말해주나?" | "이 경험에서 배운 게 뭔가요?" |
| 패턴 찾기 | 개별 감정에서 공통점 찾기 | "불안·흥미·답답함의 공통점은?" |
| 메타 관찰 | 한 걸음 물러서서 봄 | |

**예시**:
```
A, B, C의 감정 → "우리는 정보 부족에서 나오는 다양한 반응을 했네"
→ 인사이트: "사전 커뮤니케이션이 중요하다"
```

### ④ Decisional (결정·액션)

**목적**: 위 3단계에서 나온 **결정·액션**.

| 진행 | 퍼실리테이터 역할 | 예시 질문 |
|------|-----------------|---------|
| 합의 도달 | "그럼 우리가 뭘 하기로 할까?" | "이 배움에 기초해서 어떻게 할 거야?" |
| 액션 정의 | Who·What·When | "누가, 뭘, 언제까지?" |
| 커밋 | "동의하나?" 확인 | |

## ToP ORID 사용 예시: 프로젝트 사후 평가

```
🕐 Objective (5분)
Q: "이번 프로젝트에서 뭐가 일어났나요?"
A: "발표 예정일이 2주 밀렸다"
A: "고객이 중간에 요구사항을 바꿨다"
A: "팀 내 역할이 불명확했다"

🕑 Reflective (5분)
Q: "이 상황에 어떤 기분이 들었나요?"
A: "스트레스 받았어요"
A: "배운 게 많아서 보람찼어요"
A: "실망했어요"

🕒 Interpretive (10분)
Q: "왜 이런 일이 일어난 걸까요?"
Q: "우리한테 이게 뭘 말해주나요?"
A: "사전 요구사항 정의가 부족했다"
A: "프로젝트 초기에 역할을 명시해야 한다"
→ 공통 인사이트: "구조가 필요하다"

🕓 Decisional (5분)
Q: "다음 프로젝트에서 우리가 뭘 할 거야?"
A: "초기 요구사항 정의에 2주 할애하자"
A: "RACI 매트릭스로 역할을 명시하자"
→ Who: PM | What: 요구사항 정의 template 작성 | When: 다음 프로젝트 시작 전
```

**이 모델이 강력한 이유**: 사람의 뇌가 Objective → Reflective → Interpretive → Decisional 순서로 생각하게 진화했기 때문. 이 순서를 따르면 자연스럽게 깊은 의사결정에 도달.

## 퍼실리테이션 기술: Visioning & Consensus Building

### Visioning (미래상 그리기)

**언제**: 전략 수립, 조직 변화 필요할 때

**진행**:

```
① 현재 상태 진단 (15분: 지금은?)
② 미래 이상적 상태 상상 (20분: 3년 후 우리는?)
③ 그 사이의 갭 분석 (10분: 뭘 해야 그곳에 갈까?)
④ 액션 계획 (10분: 내가 뭘 할 건가?)
```

**주의**: "현재 비판" → "미래 이상화" 순서 중요. 반대로 하면 현실적이지 않게 됨.

### Consensus Building (합의 형성)

**언제**: 모두의 동의가 필요한 결정. 투표보다 강한 결정이 필요할 때.

**프로세스**:

```
① 의견 수집 (라운드로빈, 모두가 발언)
② 공통점 찾기 (어디가 겹치나?)
③ 차이점 명시 (어디가 달라?)
④ 통합 안 제시 (이 차이를 모두 포함한 안이 있을까?)
⑤ 재확인 (이 통합안에 동의하나?)
```

**예시**: 회사 핵심 가치 정의

```
개인 의견:
- A: "성실이 가장 중요"
- B: "혁신이 가장 중요"
- C: "팀웍이 가장 중요"

공통점: 모두 "조직의 성공"을 원함
차이점: 성공의 경로가 다름

통합안: "성실한 태도로 함께 혁신한다"
→ 3가지 모두 포함
```

## 퍼실리테이션 실패 패턴 & 처방

| 실패 신호 | 원인 | 처방 |
|----------|------|------|
| "모두가 조용해요" | Psychological Safety 부재. 리더가 답을 가지고 있는 것처럼 보임 | "여기서는 틀린 답이 없습니다" 명시. 리더가 먼저 불완전한 의견 제시 |
| "한두 명만 말해요" | Status Effects. 리더나 전문가만 발언 권력 있음 | 역순 발언(아래부터). Written-first 프로토콜 |
| "논쟁으로 변해요" | 이성 영역만 활성화. 감정이 없음 | Reflective 단계 추가. "이게 불편하죠?" 감정 명시화 |
