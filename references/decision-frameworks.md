# Decision Frameworks — 의사결정 프레임 상세

회의의 의사결정 구조를 정하는 것은 회의 효율의 50%. 상황에 맞는 프레임을 선택하지 못하면, 아무리 좋은 퍼실리테이션도 무의미.

---

## RAPID (Bain & Company)

**언제**: 조직 차원의 의사결정. 리소스 배분, 전략 선택, 대규모 변화.

**5개 역할 (5개 로 줄여야 함. 절대 중복 불가)**:

| 역할 | 정의 | 책임 |
|------|------|------|
| **R (Recommend)** | 제안자 | 1명만. 옵션 분석 후 권장안 제시. 팀 합의 위임받음. 독재자 아님. |
| **A (Agree)** | 동의자 | 보통 1~2명. 제안에 동의하면 구현에 참여. 거부권은 자신의 영역 내에만. |
| **P (Perform)** | 실행자 | 결정 후 구현 담당. 비용/일정/품질 책임. 회의 중 결정 재논의 불가. |
| **I (Input)** | 정보제공자 | 여러 명 가능. 전문지식·데이터 제공. 의사결정은 아님. |
| **D (Decide)** | 최종결정자 | 1명만. 모든 의견 청취 후 최종 판단. 책임을 져야 할 위치. |

**진행 프로세스**:

1. R이 분석 → 옵션 정리 (3개 이하)
2. I들이 각 옵션의 장단점 피드백
3. R이 권장안 + 근거 제시
4. A들이 "내 영역에서 구현 가능한가?" 판단
5. D가 최종 결정 + 근거 기록

**핵심 규칙**:
- **R과 D는 다른 사람이어야 함**. (같으면 독재)
- 결정 후 P가 "다시 생각해봤는데" 불가능. 결정은 최종.
- Agree자가 거부 → D가 최종판단. A의 거부권은 자신의 영역 내에만 적용.

**장점**: 명확한 책임. 빠른 결정.
**단점**: 정보비대칭 시 R의 권력 과대. 일부 A가 "동의했지만 내가 원한 게 아니" 느낄 수 있음.

**한국 조직에서의 팁**: D는 보통 리더. R은 해당 영역 전문가. A는 이해관계자. 이 구조가 안 정해지면 회의는 엉망.

## Sociocracy 3.0 — Consent Decision Making

**언제**: 팀 내 정책/프로세스 변화. "No reasoned objections"가 핵심 — 100% 동의가 아니라 "거칠 이유가 없다"는 의미.

**핵심 개념**:
- **동의 ≠ 열정적 찬성**. "나는 이게 최선이 아니라고 생각하지만, 거절할 이유가 없다" = 동의.
- **거칠 만한 이유 = reasoned objection**: 논리적 근거가 있어야 함. "나는 이게 싫어" 같은 감정은 거칠 이유가 아님.

**진행 프로세스**:

1. **Proposal**: 제안자가 자신의 안 제시
2. **Clarifying questions**: 이해 안 되는 부분만 질문 (논의 아님)
3. **Reactions (go-around)**: 모두가 한 번씩 발언 (토론 X, 개인의견만)
4. **Amend & clarify**: 제안자가 피드백 받아서 수정할 부분 수정
5. **Objection round**: 모두가 "나는 이 제안에 거칠 이유가 있다"고 명시적으로 선언
6. **Integration (필요시)**: 거절자와 제안자가 함께 수정안 만듦
7. **Consent**: 거칠 이유가 없으면 동의. 시행.

**거칠 이유의 조건**:
- 논리적 근거가 있어야 함
- "이 제안이 실행되면 뭔가 안 될 것 같다"는 구체적 우려
- 개인적 선호도는 거칠 이유 아님

**장점**: 모두의 목소리 반영. 거칠 만한 이유는 설득하게 함 (innovation 유도). 빠른 결정 (합의 찾기보다 거칠 이유 찾기).
**단점**: 거칠 이유의 기준이 모호할 수 있음. 한 명의 거칠 이유가 전체 계획 지연 가능.

**한국 조직에서의 팁**: "거칠 이유가 있으면 손 들어"를 명시적으로 확인. 침묵은 동의로 간주. 비판 문화가 약한 조직에서는 거칠 이유가 나오지 않을 수 있으니, 사전에 "이 부분이 위험할 것 같은 사람?" 식으로 프롬프트 필요.

## Advice Process (Teal Organization)

**언제**: 권한위임된 팀에서 개인이 결정. "누구든 의사결정 가능하지만 조언은 반드시 구해야 함."

**진행 프로세스**:

1. 누군가 결정이 필요 → 자신이 결정 권한 있는지 확인
2. 전문가 + 영향받는 사람들에게 **조언 구하기** (회의 아님, 개인면담)
3. 조언을 들었지만 **최종 결정은 본인**
4. 결정 공유 (결과 책임은 본인)

**핵심**: 모두의 동의 필요 없음. 조언은 참고만. 근거 있는 결정이면 실행.

**예시 (AES): 팀원이 새로운 프로세스 도입을 결정하려면**:
1. 관련 전문가 2~3명 개인면담으로 조언 구하기
2. 영향받는 사람들에게 "이렇게 하려는데 문제 있을까?" 구하기
3. 모인 조언 검토 후 "내가 이렇게 결정한다" 선언
4. 실행. 실패하면 본인 책임.

**장점**: 빠른 의사결정. 권한위임 강화. 개인 책임감 높음.
**단점**: 리더와 조언자의 관계 신뢰 필수. 조언 무시하고 실패하면 신뢰 깨짐. 조직 내 신뢰 수준이 높아야 작동.

**한국 조직에서의 팁**: 명확한 책임 범위 정의 필수. "너는 이 영역에서 결정 가능하다"는 권한이 명시되어 있어야 함. 없으면 "혼자 결정하나요?" vs "조언만 듣나요?" 혼동 발생.

## Amazon Silent Reading + 6-Page Memo

**언제**: 복잡한 전략 의사결정. 시간이 충분할 때(긴급 결정 X).

**프로세스**:

1. **사전 준비**: 제안자가 6페이지 메모 작성 (PowerPoint 금지, 이야기 형식의 산문)
2. **회의 시작**: 모두가 30분 동안 조용히 읽기 (토론 X)
3. **Q&A**: 명확화 질문만 (토론 X)
4. **토론**: 이제 의견 교환
5. **결정**: 리더가 판단

**6페이지 메모의 구조**:
- 배경 (상황)
- 문제 정의 (왜 결정이 필요한가)
- 선택지 분석 (장단점)
- 권장안 + 근거
- 예상 결과

**왜 PowerPoint 금지?**: 슬라이드는 정보 부족 → 청자의 상상력에 맡김 → 모두 다르게 이해. 산문은 논리적 흐름 → 모두 같은 정보량으로 같은 수준의 판단 가능.

**장점**: 정보 비대칭 제거. 리더/목소리 큰 사람의 영향 최소화. 깊은 이해.
**단점**: 시간 소요. 메모 쓰는 시간 + 읽는 시간 + 토론 = 3~4시간 필요.

**한국 조직에서의 팁**: "읽는 시간 동안 메모 금지"를 강제. 한국 조직은 읽으면서 질문하려고 하거나 "이 부분 이해 안 됨" 말하고 싶어 함.

## Toyota Nemawashi + Ringi System

**언제**: 구현 변수가 많은 대규모 결정 (조직문화 변화, 프로세스 개선). 실제로 실행할 사람들의 동의가 중요할 때.

**Nemawashi (根回し)**: 의사결정 **전**에 이해관계자들을 개별 만남으로 사전 설득.

1. 의도/안을 설명
2. 상대의 우려 경청
3. 수정 → 다시 만남
4. 반복

→ 공식 회의에 들어가기 전에 이미 대부분 동의된 상태

**Ringi (稟議)**: 공식적인 결정 결재 순환 문서.

1. 최하위 담당자가 작성 → 결재 올림
2. 각 부장급이 도장 → 다음 사람
3. 최상위 리더 서명
4. 공식 결정

**장점**: 구현할 사람들이 이미 동의 → 저항 최소. 중간관리자들이 이미 생각해봤으므로 실행 문제 사전 파악.
**단점**: 엄청 느림. 3~4주 소요 가능. 합의 문화가 강한 조직에서만 작동.

**한국 조직에서의 팁**: Nemawashi는 좋지만 Ringi처럼 순환 결재는 시간 낭비 우려. 대신 "핵심 이해관계자 사전 면담" 단계만 도입하는 것도 효과적.

## Bridgewater Radical Transparency

**언제**: 메리토크래시 문화. 모든 의견이 동등하게 평가되는 조직.

**프로세스**:

1. 모든 회의 녹음
2. 회의 중 실시간 투표 (believability-weighted)
3. 의견의 정확성 기반으로 각자의 "신뢰도" 점수 누적
4. 신뢰도가 높은 사람의 투표가 더 큰 무게

**예**: 전략 회의에서 A와 B가 다른 의견 → 투표 시 A의 신뢰도가 90%면 A의 표는 0.9로 계산, B의 신뢰도가 60%면 B의 표는 0.6으로 계산.

**장점**: 계급이나 직급이 아닌 **신뢰도 기반** 의사결정. 좋은 아이디어가 주니어에서 나와도 그대로 반영.
**단점**: 신뢰도 산정이 복잡. 조직 문화가 매우 개방적이어야 (비판에 강해야 함). 한국 조직에서는 거의 불가능.

**한국 조직에서의 팁**: 그냥 참고만 하기. "신뢰도 기반 투표"는 도입 불가. 대신 "직급 상관없이 의견 청취" 정도만 도입.

## 의사결정 프레임 선택 매트릭스

| 상황 | 추천 프레임 | 이유 |
|------|-----------|------|
| 조직 전략 결정 (리소스 큼) | RAPID | 책임 명확. 빠른 결정. |
| 팀 프로세스 변화 | Sociocracy Consent | 팀 구성원 모두의 거칠 이유 확인 필수. |
| 권한위임된 팀의 개인 결정 | Advice Process | 개인 책임감 + 전문성 활용. |
| 복잡한 전략 (시간 충분) | Amazon Memo | 정보 비대칭 제거. 논리적 검증. |
| 구현이 어려운 대규모 변화 | Nemawashi | 사전 설득으로 저항 최소화. |
| 아주 긴급 (1시간 내 결정) | RAPID | 단순하고 빠름. |
| 팀 응집력이 낮음 | Consent | 거칠 이유를 통해 우려사항 명시화. |
| 신뢰 문화가 강함 | Advice Process | 개인 책임 유도. |
| 신뢰 문화가 약함 | RAPID | 리더의 판단이 명확. 책임도 명확. |

## 프레임 조합 예시

**예시 1: 신입사원 정책 변경**

상황: HR팀이 신입사원 온보딩 프로세스 개선
시간: 2주 충분
결정 방식: Nemawashi + Sociocracy

1. HR 담당자 → 각 부서 리더 개별 미팅 (Nemawashi)
2. 피드백 받아서 수정
3. 최종안으로 전사 회의에서 Consent 프로세스 (거칠 이유 확인)

**예시 2: 해외 시장 진출**

상황: 이사회에서 신시장 진출 여부 결정
시간: 1개월 충분
결정 방식: Amazon Memo + RAPID

1. 전략팀 → 6페이지 메모 작성
2. 이사회 전원 30분 조용히 읽기
3. Q&A → 토론
4. CEO(D)가 최종 결정

**예시 3: 팀 내 업무 분담**

상황: 디자인 팀 내 프로젝트 배분
시간: 긴급 (내일)
결정 방식: Advice Process

1. 팀 리더가 최적 배분 생각
2. 각 팀원 개별로 조언 구하기
3. 의견 반영해서 최종 결정 및 공시

## 결정 후 기록 (모든 프레임 공통)

| 항목 | 기록 |
|------|------|
| **결정** | 정확히 무엇을 결정했는가 (1문장) |
| **의사결정 프레임** | RAPID / Consent / Advice 중 뭘 썼는가 |
| **의사결정자** | 누가 최종 결정했는가 (1명 명확히) |
| **근거** | 왜 이렇게 결정했는가 (2~3문장) |
| **영향범위** | 누가/어디가 영향받는가 |
| **액션** | 이 결정을 구현하는 데 누가 뭘 해야 하는가 |
| **기한** | 언제까지 구현되어야 하는가 |

**이 기록을 회의록에 구조적으로 분리하면, 6개월 후 "그때 왜 이 결정을 했지?" 물을 때 학습자료가 됨.**
