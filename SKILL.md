# Meeting Engine — 회의 전주기 엔진

ceo-pipeline이 "무엇을 어떤 순서로 할 것인가", management-skill M3가 "실행 시스템을 어떻게 굴리는가"라면, 이 스킬은 **"사람이 모이는 순간을 최대 효율로 바꾸는 것"**.
회의는 조직의 가장 비싼 자원(시간×인원수). 나쁜 회의는 조직을 죽이고, 좋은 회의는 의사결정 속도를 가속한다(Perlow, HBR).

---

## ⛔ 절대 규칙

| # | 규칙 | 이유 |
|---|------|------|
| 1 | **목적 미확정 회의 거부** — "왜 모이는가"에 대한 답 없으면 회의 설계 진행 불가 | 목적 없는 회의 = 시간 낭비의 정의 |
| 2 | **의사결정 ≠ 정보공유 ≠ 브레인스토밍 구분 필수** — 유형별 구조가 완전히 다름 | 혼합하면 아무것도 안 됨 |
| 3 | **액션아이템 = Who + What + When** — 3요소 중 하나라도 빠지면 미완성 | "검토하겠습니다"는 액션아이템이 아님 |
| 4 | **시간 배분 사전 설계 필수** — 아젠다 항목별 분 단위 배분 | 파킨슨 법칙: 시간 제한 없으면 논의는 무한 팽창 |
| 5 | **결정사항 명시적 기록** — 누가 무엇을 결정했는지 회의록에 구조적으로 분리 | 결정이 묻히면 다음 회의에서 같은 논의 반복 |
| 6 | **집단 병리 사전 스크리닝 필수** — 설계 전 Groupthink·Abilene Paradox·Social Loafing 위험 진단 | 위험을 모르면 구조로 방어할 수 없음 |

## §3. 집단 병리 스크리닝 (설계 전 필수)

→ references/group-pathology.md 상세 프로토콜

설계 모드 진입 **전** 반드시 아래 7가지 병리를 현재 그룹에서 스캔하고 방어 구조를 설계에 반영.

| 병리 | 신호/위험도 | 설계 시 방어 | 참고 |
|------|-----------|----------|------|
| **Groupthink** | 높은 응집력+독재적 리더+외부 압력 → 비판 자가검열 | 명시적 devil's advocate, 익명 의견수렴, 외부 전문가 초청 | Janis 8증상 |
| **Abilene Paradox** | 각자 동의하지 않는데 침묵→집단오류 | 사전 익명 선호도 폴링, 명시적 "나는 이렇게 생각" 발언 | Harvey |
| **Production Blocking** | 한 명 말할 때 남들이 생각 못함 → 아이디어 20-30% 손실 | 개별 사고+공유의 분리, Nominal Group, 비동기 아이디어 입력 | Diehl & Stroebe |
| **Social Loafing** | 그룹 커질수록 개인 노력 ↓ | 개인책임 명확화, 소규모 그룹(4-6명), 기여도 가시화 | Latané |
| **Anchoring** | 첫 정보가 판단 지배 | 다중 앵커 동시제시, Explict de-anchoring, 동시 공개 투표 | Kahneman |
| **Status Effects** | 상위직이 더 말함 → 정보 편향 | 역순 발언순서, 동료가 먼저, 역할 명시적 분리 | Karpowitz & Mendelberg |
| **Noise** | 피로·기분·동기 차이 → 판단 불일치 | Decision Hygiene: 독립판단→집계→토론, 프로세스 표준화 | Kahneman, Sibony, Sunstein |

**스크리닝 결과**: 고위험 병리 2개 이상 감지 시 § 절대규칙 #6 발동 → 설계에 방어 구조 추가.

## §4. 실행 파이프라인

### 설계 모드

```
①목적·유형 확인 
→ ②집단 병리 스크리닝 
→ ③의사결정 프레임 선택 
→ ④퍼실리테이션 패턴 라우팅 
→ ⑤아젠다 설계 
→ ⑥사전 준비물 정의 
→ ⑦진행 스크립트
```

**①목적·유형**: 형에게 "이 회의의 목적이 뭔가요?"부터. 단문 명확화 필수. 혼합 유형은 블록 분리.

**②집단 병리 스크리닝**: §3 테이블. 현재 그룹의 응집력·리더십 패턴·외부 압력 확인 → 고위험 병리 2개 이상 시 방어 구조 설계.

**③의사결정 프레임**: §2 매트릭스 + → references/decision-frameworks.md 상세. 조직문화, 긴급성, 영향범위에 따라 RAPID/Consent/Advice/Consensus 선택.

**④퍼실리테이션 패턴**: → references/facilitation-science.md 참고. Kaner Diamond(Divergent→Groan→Convergent 단계), Liberating Structures(33가지), ToP ORID 모델 중 선택 라우팅.

**⑤아젠다 설계**:

```
[회의명] — [날짜] [시간] [장소/링크]
목적: [1문장]
유형: [§2에서 매핑]
참석: [이름(역할)]
의사결정프레임: [§3에서 선택]
병리방어: [고위험 병리→구조적 방어]

[시간] 아젠다 항목 1 — [담당자] — [유형: 정보/논의/결정]
  목표: [이 항목이 끝날 때 달성할 것]
  사전준비: [읽어올 것]

[시간] 아젠다 항목 2 — [담당자] — [유형]
  목표: [...]

[시간] 액션 확인 + 마무리 — [주재자]
```

**시간 배분 원칙**: 의사결정 항목에 전체의 60%+ 배분. 정보공유는 최소화(사전 공유로 대체). 마무리 5분 필수(액션아이템 확인).

**⑥사전 준비물**: 각 아젠다 항목별 "참석자가 미리 읽어와야 할 것" 정의. 사전 공유 없이 회의에서 처음 보는 자료 = 회의 시간 낭비.

**⑦진행 스크립트**: 주재자용 진행 가이드 (→ references/facilitation-science.md 상세 프로토콜).

| 단계 | 주재자 행동 | 시간 |
|------|-------------|------|
| 오프닝 | 목적·아젠다·기본규칙 확인. 병리방어 구조 간단 공지(필요시) | 2분 |
| 각 아젠다 | 담당자 발표 → Kaner Diamond 진행(Divergent→Groan→Convergent) | 배분 시간 |
| 타이머 | 각 항목별 타이머. 초과 시 "파킹랏에 넣고 다음" 또는 "5분 연장" 판단 | — |
| 마무리 | 결정사항·액션아이템 낭독 + 담당자·기한 명시 확인 | 5분 |

### 정리 모드

```
①회의 노트 입력 → ②구조화 → ③결정 근거 추출 → ④액션아이템 검증 → ⑤공유용 회의록
```

**②구조화**: 날것 노트를 아래 템플릿으로 변환.

```
# 회의록: [회의명]
일시: [날짜 시간] | 참석: [이름]
의사결정프레임: [RAPID/Consent/Advice/...]

## 결정사항
- [결정 1]: [내용] — 결정자: [이름] | 근거: [왜 이렇게 결정했나]
- [결정 2]: [내용] — 결정자: [이름] | 근거: [...]

## 액션아이템
| # | 액션 | 담당 | 기한 | 상태 |
|---|------|------|------|------|
| 1 | [구체적 행동] | [이름] | [날짜] | ⬜ |

## 논의 요약
[아젠다 항목별 핵심 논의 내용. 2~3문장으로 압축.]

## 파킹랏 (다음에 다룰 것)
- [미결 이슈 1]
```

**③결정 근거 추출**: 각 결정 옆에 "왜 그렇게 결정했나" 기록. 후속 회의에서 재논의 방지 + 학습자료 축적.

**④액션아이템 검증**: Who(담당자) + What(구체적 행동 — "검토" ✗, "A 문서의 B 섹션을 C 관점에서 수정하여 D까지 완료" ○) + When(기한 — 날짜). 3요소 미충족 시 형에게 보완 요청.

### 시스템 모드

```
①현행 회의 체계 진단 → ②비효율 패턴 매칭 → ③병리 근인 분석 → ④개선안 설계 → ⑤전환 로드맵
```

→ references/meeting-systems.md 상세 진단 프로토콜

**회의 비효율 7대 패턴**:

| # | 패턴 | 신호 | 근인(행동과학) | 처방 |
|---|------|------|---------------|------|
| 1 | 목적 부재 | "이 회의 왜 하는 거지?" | 역할 정의 부재 + 습관 회의 | 목적·유형 재정의. 불필요 시 폐지 |
| 2 | 참석자 과다 | 10명+ 의사결정 회의 | 신호(Status Effects) → 의견 편향 | RAPID 역할 분류. 불필요 참석자 → 비동기 |
| 3 | 아젠다 부재 | 시작하면서 "오늘 뭘 할까?" | Groupthink: 사전 구조 없음 → 강자 의견 추종 | 24시간 전 아젠다 배포 필수화 |
| 4 | 시간 초과 습관 | 매번 30분 연장 | Parkinson Law: 제약 부재 → 시간 팽창 | 타이머 + 파킹랏 도입 |
| 5 | 결정 미기록 | "그래서 뭘로 결정했었지?" | 의사결정자 불명확 → 근거 불명 | 실시간 결정 로그 + 마무리 낭독 + 근거기록 |
| 6 | 액션 미추적 | 지난 회의 액션 확인 안 함 | Accountability 부재 → 동기저하(Social Loafing) | 회의 오프닝에 지난 액션 리뷰 5분 + 추적표 |
| 7 | 회의→회의 체인 | "이건 다음 회의에서" 반복 | Production Blocking: 동기식 시간 비낭비 + Bias for action 부재 | 비동기 해결 기준(Async vs Sync Decision Framework) 적용 |

## §5. 회의 유형별 진행 템플릿

### 의사결정 회의

| 단계 | 시간비중 | 내용 | 주재자 기술 |
|------|----------|------|----------|
| 맥락 공유 | 10% | 사전 배포 자료 기반 1분 요약 | "이 결정이 왜 중요한지 30초 요약" |
| 선택지 제시 | 20% | 2~3개 선택지 + 각 장단점 | 선택지의 장단 동등하게 프레임 (Anchoring 방지) |
| 질의·논의 | 40% | 구조화된 질의 → 핵심 쟁점 수렴 (Kaner Diamond) | Divergent(다양성 환영) → Groan(불편함 견디기) → Convergent(수렴) |
| 결정 | 20% | 의사결정자 판단 + 근거 명시 | "이 근거로 X를 선택합니다" |
| 액션 확인 | 10% | Who·What·When 확정 + 의존성 맵 | "누가 뭘 언제까지? 다음 회의는?" |

### 진척리뷰 (스탠드업/주간)

→ references/meeting-systems.md "Scrum Ceremonies" 상세

| 단계 | 시간비중 | 내용 | 방지할 병리 |
|------|----------|------|-----------|
| 지난 액션 리뷰 | 20% | 완료/미완료/블로커. 미완료 시 근인 | Social Loafing: 개인책임 명확화 |
| 지표 체크 | 20% | metric-tracker 대시보드 기반 | Status Effects: 모두가 발언할 기회 |
| 이슈 공유 | 30% | 블로커·리스크만. 해결은 별도 | Production Blocking: 시간 제약 명확 |
| 의사결정 (있으면) | 20% | 이슈 중 즉시 결정 가능한 것만 | Abilene Paradox: 명시적 폴링 |
| 다음 주 초점 | 10% | 핵심 우선순위 1~3개 | Noise: 개인별 해석 다름 → 문서화 |

## Gotchas

| 함정 | 대응 |
|------|------|
| 목적 없이 아젠다 작성 | 절대규칙 #1. "왜 모이나?" 먼저 |
| 병리 감지 안 함 | 절대규칙 #6. §3 스크리닝 체크리스트 사용 |
| 의사결정+브레인스토밍 혼합 | §2 유형 분리. 블록별로 나누거나 별도 회의 |
| "검토하겠습니다" 액션 | Who+What+When 3요소 필수. "검토"→"A를 B 기준으로 평가하여 C까지 결과 공유" |
